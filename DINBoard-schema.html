02

Warstwy kodu — co gdzie żyje

DINBoard trzyma ścisłego MVVM-podobnego podziału. Komponenty tylko renderują, hooki orkiestrują stan, lib/ liczy domenę, types/ deklaruje kontrakty. Każda warstwa ma swoje may / may not — złamanie tej reguły to dług techniczny.

UI
src/components/**
Tylko renderowanie. Zero obliczeń domenowych. Konsumuje dane z lib/.
Orchestracja
src/hooks/**
Stan UI, debounced persist, koordynacja akcji. Persistencja tylko przez lib/.
Domena
src/lib/**
Wszystkie obliczenia: fazy, walidacja, geometria, eksport PDF, parsowanie SVG, I/O pliku.
Kontrakty
src/types/**
Tylko typy i normalizery. Nigdy obliczenia.
Desktop
src-tauri/**
Integracja OS (Tauri). Nigdy domena.

State flow — diagram przepływu

Każda akcja użytkownika przepływa przez ściśle określone warstwy. UI wysyła intencję → hook ją orkiestruje → lib liczy domenę → store zapisuje. Poniżej: diagram pełnego cyklu + 3 najczęstsze wzorce akcji.

Tokeny wizualne — ciemny motyw
