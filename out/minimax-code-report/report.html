SECTION 01 · MISSION

A workspace built for shipping, not chatting.

Most AI coding tools stop at autocompletion or chat. MiniMax Code starts there and keeps going — into a runtime that can read your repository, run tools, write back changes, persist across sessions, and verify its own work. The end goal isn't a clever answer. It's a commit you can ship.

01What problem we solve

Engineering teams don't need another autocomplete. They need a teammate that can take a ticket — read the codebase, follow the conventions, run the tests, hand back a patch — and stay coherent across days, not just turns. MiniMax Code is the workspace that hosts that teammate. The product blends a foundation model, an agentic harness, and a tool-rich runtime behind a single command surface.

02Three beliefs we ship with

⏵ Agents must have memory

Stateless chat loses context on every reload. MiniMax Code treats memory as a first-class artifact — three layers (project, agent, user) — and writes to it as the agent works, not after.

⏵ Tools belong in the harness

The model can think, but the harness is what reads files, runs shells, calls APIs. The harness wraps the model with the same rigor an OS wraps a process.

⏵ Shipping beats impressive

We optimize for diffs that land, not chat that wows. Every capability ships with a verification path — tests, lint, build — before the agent hands a result back.

⏵ Independence from any one model

Model quality will keep moving. The harness is the long-term asset; the model is a swappable component behind it.

POSITIONING SUMMARY
MiniMax Code is to chat-based coding assistants what a CI runner is to a script: not smarter syntax — a different unit of work. The unit isn't an answer; it's a verified change.

SECTION 03 · ARCHITECTURE

Five layers, one runtime.

MiniMax Code is structured as five horizontal layers. Each layer has its own contract and can be swapped independently — the foundation model is replaceable; the harness is the long-term asset.

SURFACE TUI · CLI · IDE extension · Web · CI runner User-facing
TOOL LAYER Read · Write · Edit · Bash · Glob · Grep · MCP tools · Custom tools Sandboxed
HARNESS Agent loop · Memory · Skills · Hooks · Sessions · Permissions Long-term IP
MODEL LAYER MiniMax M3 (primary) · secondary models · per-task routing Swappable
PLATFORM Node 22+ · multi-OS · terminal-native · offline-capable Stable

SECTION 04 · WORKFLOWS

How teams actually use it.

Four autonomy levels define how much of the loop the agent owns. Most teams settle into a mix — heavy on Ask in code review, heavy on Build in greenfield work — and that's by design.

01The four autonomy levels

Level Name What the agent does Best for Risk
L1 Ask Read-only Q&A, explanations, plan proposals Code review, onboarding Low
L2 Plan Reads files, drafts diffs, asks before applying Refactors, migrations Low
L3 Build Applies diffs, runs tests, iterates until green Feature scaffolding Med
L4 Yolo Full loop, broad permissions, batches tasks Spike / throwaway work High

03Three entry paths

Entry

Solo developer

  • Install CLI, point at a repo
  • Use Ask / Plan only
  • Let the agent draft PRs for review
  • Zero infra changes
Recommended

Product team

  • Shared agents per domain (frontend, infra, data)
  • Skills per repo (test conventions, deploy)
  • L2 / L3 in feature work, L1 in review
  • CI integration for auto-triage
Power

Platform engineering

  • Custom tool/MCP servers
  • Hook-driven policy enforcement
  • Custom model routing per task
  • Audit pipeline to internal SIEM
DAY-TO-DAY PATTERN
Most teams converge on L2 / L3 in the IDE, L1 in code review, and L4 only on weekends. The workspace treats autonomy as a per-task decision, not a global flag.
SECTION 05 · TRUST

Safe-by-default, auditable, portable.

An agent that ships code to production needs stronger guardrails than a chatbot. MiniMax Code treats every tool call as a policy event: permissioned by default, logged in full, and replayable from the audit trail.

03Three conclusions for trust

⏵ 01
Permissioned by default
Every tool starts permissioned. The grant is opt-in, scope-narrow, and revocable. No ambient authority inherited from a previous session.
⏵ 02
Auditable by design
Every action produces a structured event: actor, target, intent, result. Replay any session from its log; reproduce any failure.
⏵ 03
Recoverable from bad runs
Checkpoints, undo, branch-and-merge worktrees. A bad agent run is not a fire — it's a tab you close and re-open at the last green.
⏵ INSIGHT
For distribution, auditability is the make-or-break capability. Teams will not adopt a coding agent that ships without a verifiable trail. MiniMax Code is built around that requirement, not around it.

03Three takeaways

01
Memory is the moat.
Three layers of memory — project, agent, user — outlast every model swap. Five years from now, the most valuable artifact in the workspace is the memory store, not the model that wrote to it.
02
Verification is the unlock.
Agents that close the loop with their own tests move from "helpful" to "trusted". The integration with CI is the difference between a demo and a deployment.
03
Teams beat solo agents.
A single agent never has all the context; a coordinated team of agents, each with a tight role, scales further than any one mega-prompt.
